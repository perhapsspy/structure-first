# Skill: Structure First (Korean Pair)

> 영문 기본 문서: `SKILL.md`

## Purpose

> **Primary Flow**: 로직을 위에서 아래로 읽히게 만드는 메인 성공 경로

코드 생성, 리팩터링, 리뷰 시 **읽히는 성공 흐름(Primary Flow)**을 최우선으로 둡니다.
경계는 필요한 만큼만 두고, **역할 하나와 명확한 I/O를 가진 작은 단위(Atom)**로 조합합니다.
테스트는 구현 추종이 아니라 **계약 중심(contract-driven)**으로 안정성을 확보합니다.

## When to Use

- 이 스킬은 **실제 코드 구조 작업**에 우선 적용한다: 구조 리팩터링, 코드 형태 변경, boundary 단순화, 판단 규칙 소유 정리, 구조 중심 코드 리뷰.
- 변경 분류, 커밋 계획, 범위 분석처럼 구현을 직접 바꾸지 않는 작업에서는 이 스킬을 내부 판단 기준으로만 활용하고, structure-first 응답 템플릿은 적용하지 않는다.
- 코드가 위에서 아래로 자연스럽게 읽히지 않을 때
- 함수/모듈 분해가 과도해지고 유틸이 늘어날 때
- 테스트가 구현 세부를 따라가려 할 때

## Do Not Use

- 1회성 실험/탐색 코드
- 아주 작은 수정인데 구조 개입이 과해지는 경우

## Priorities (Bias)

- 읽히는 흐름 > 구조적 단순성 > 재사용성 > 추상화
- 미래 가능성보다 **현재 코드의 명확성** 우선
- 분해는 목적이 아니라, 읽힘이 좋아질 때만 수행

## Scale Application

- 어떤 스케일에서도 원칙은 같다.
- 먼저 **현재 작업 단위(current unit)** 를 고정한다.
- 로컬 수정은 함수/파일, 기능 작업은 모듈/유스케이스, 더 큰 리팩터링은 capability/subsystem 단위로 본다.
- 그 단위에서 Primary Flow를 읽히게 만든다.
- 함수/파일 스케일에서는 얕게 멈춘다.
- 여러 boundary 단계가 한 흐름에 섞이기 시작하면 그 단계는 이름을 준다.
- 좋은 결과는 여전히 `normalize -> load -> decide -> return` 정도로 읽히는 경우가 많다.
- 분기 설명이 중간 데이터 구조나 도우미 함수 체인에 기대기 시작하면, 현재 위치에 더 남기거나 더 내려가지 않는다.
- capability/subsystem 스케일에서는 진입점과 주 오케스트레이터를 드러낸다.
- 상태와 판단 규칙 소유자를 분명히 해서 하위 단위가 정책을 다시 계산하거나 같은 외부 관찰 결과를 다른 경로로 다시 쓰지 않고 평가된 결과를 소비하게 만든다.

## Work Routine

1. **Intent 고정**
- 변경/코드가 하는 일을 한 문장으로 명시
- 구조를 바꾸기 전에 현재 작업 단위를 먼저 명시: 함수/파일, 모듈/유스케이스, capability/subsystem

2. **Boundary 최소화**
- 단계를 I/O, 도메인 판단, 변환으로 구분
- 경계는 필요한 만큼만 유지

3. **Primary Flow 먼저**
- 성공 흐름이 위→아래로 한 번에 읽히게 구성
- 분기/예외는 메인 흐름을 끊지 않게 처리(early return 또는 아래로 내리기)
- 로컬 분기 로직을 평평하게 만들 때는, 규칙이 누적형인지 우선순위형인지 보존한다

4. **Atom 도출**
- Primary Flow 문장이 함수 또는 하위 단위로 분리될 때 더 읽히면 분리
- Atom은 현재 작업 단위 기준에서 역할 하나와 명확한 I/O를 가진다
- 가능하면 순수 함수 선호
- 함수/파일 스케일에서는 분해가 로컬 흐름을 더 잘 설명하지 못하면 멈춘다. 현재 위치에 더 남기거나 현재 작업 단위를 다시 선언한다

5. **Composition 한 곳**
- 오케스트레이션은 현재 작업 단위에서 한 위치에서만 수행
- Atom 간 직접 의존/호출 최소화

6. **Side Effects를 Boundary로**
- I/O, 상태 변경 같은 부작용은 경계에 모음
- 내부 로직은 계산/판단 중심으로 유지

7. **Read Order 정렬**
- 파일 레벨에서는 위에서 아래로 읽히게 하기 위해 기본적으로 export/public -> orchestrator -> atoms -> utils 순으로 둔다

8. **성장 제어**

- 공개 I/O/시그니처는 확정된 책임만 담는 최소 형태로 시작하고, 책임이 실제로 바뀔 때만 키운다(새 외부 입력, 의미 혼합, 경계 이동).
- 기본적으로 비즈니스 판단/계산 규칙과 같은 외부 관찰 결과에 영향을 주는 write 경로는 한 unit에서 소유한다. 여러 writer를 조정하는 것 자체가 책임이라면 그 coordinator를 명시한다.
- 비동기 경계에는 최신성/완료 정책을 소유하는 owner가 하나 있어야 한다. 여기에는 최신성 충돌을 어떻게 해소하는지와 시작/종료 책임 균형이 포함된다.
- 판단 규칙 소유가 바뀌거나 새 동등 경로를 넣는다면, 가능하면 같은 변경에서 기존 경로를 제거/비활성화한다. 어렵다면 단계적 마이그레이션 계획(소유자, 종료 조건)을 함께 남긴다.
- `판단 규칙(Decision rule)`: 동작을 결정하는 반복 조건/가중치/우선순위 규칙.
- `동등 경로(Equivalent path)`: 외부에서 관찰되는 결과가 같은 대체 실행 경로.

## Test Guidance (Unit tests for Atoms)

- 가능한 한 가장 안정적인 Atom 단위에서 테스트를 충분히 작성
- 구현 세부가 아니라 현재 작업 단위와 Atom/경계 사이의 **계약(I/O, 불변조건, 경계값)**을 검증
- 오케스트레이션이나 경계 통합이 주요 리스크라면 현재 작업 단위를 직접 테스트한다
- 비동기/상태성 경계가 있다면 stale 결과 처리, 완료 균형, 동등 입력 no-op 같은 ownership 계약을 그 경계를 소유한 가장 안정적인 단위에서 검증한다
- 이번 변경에서 테스트를 추가할 수 없다면 그 사실을 명시하고, 다음 안정적인 Atom과 필요한 계약 케이스를 함께 적는다
- 테스트 코드도 읽히게 유지: `each`/table 케이스로 중복을 줄이고, 구조를 흐리지 않는 작은 헬퍼만 허용하며, 테스트 하나는 핵심 주장 하나에 집중한다

## Stop Rules / Anti-patterns

- 분리 후 인자/상태 전달이 늘어나면 되돌리기
- 보기 좋으려고만 함수/파일 분리하지 않기(유틸 난립 금지)
- 이름이 설명문처럼 길어지면 경계를 재검토
- 미래 재사용 가정의 추상화/레이어 추가 지양
- 테스트에서 과도한 추상화/헬퍼 난립 지양
- "나중에 필요할 수도 있음" 이유로 인자 선추가 금지
- 동일 판단 규칙의 다중 소유 금지
- 여러 writer를 조정하는 것 자체가 명시적 책임이 아닌데도 같은 외부 관찰 결과를 여러 writer 위치에 나눠두지 않기
- ownership과 reset 의미가 명시적이지 않은데도 상위 경계의 상태를 로컬 가변 상태로 동기 미러링하지 않기
- 한 unit이 읽은 입력/update 경로에 다시 그 경로로 써 넣는 self-feedback loop를 만들지 않기
- 단계적 마이그레이션 계획(소유자, 종료 조건) 없이 동등한 신규/구경로 병행 유지 금지

## Final Gates

- 성공 흐름이 위→아래로 한 번에 보이는가?
- 현재 작업 단위를 분명히 적었고, 지금 변경에 맞는 단위가 유지되고 있는가?
- 함수/파일 스케일이라면 중간 데이터 구조나 도우미 함수 체인이 분기 설명을 대신하지 않고도 충분히 얕게 읽히는가?
- 분리가 실제 책임/경계 변화에 대응하는가?
- Atom I/O를 한 줄로 설명 가능한가?
- 부작용이 경계에 모여 있는가?
- 테스트가 계약 중심이며 간결한가?
- 판단 규칙이 이동했다면 소유자가 한 곳으로 명확해졌는가?
- capability/subsystem 스케일이라면 진입점, 주 오케스트레이터, 판단 규칙 소유자를 말할 수 있는가?
- 인자 증가 정당성과 구경로 정리(또는 단계적 마이그레이션 계획)가 확인되는가?

## Completion Evidence

이 형식은 실제 코드 구조 작업이나 구조 중심 리뷰를 설명하는 데 실질적으로 도움이 될 때만 사용한다.
리팩터링 작업이라도 이 형식을 기본 사용자 응답으로 취급하지 않는다.
비구현 작업에서는 사용자가 명시적으로 원하지 않는 한 응답을 자연스러운 과업 형식으로 유지한다.

형식이 실제로 유용한 경우에만 아래 4줄을 제시:

- `Current Unit:` function/file | module/use case | capability/subsystem
- `Primary Flow:` top-down 3~6줄
- `Boundaries:` I/O 경계 목록
- `Tests:` `added ...` 또는 `deferred because ...; next stable Atom(s): ...; required contract cases: ...` (해당되면 freshness/completion contract 포함)

판단 규칙 소유가 바뀌는 리팩터링에서는 아래 항목을 함께 제시:

- `Decision Ownership:` `rule -> owner unit`; 중복 소유 제거 여부 yes/no

시그니처/경계가 커졌거나 구경로를 대체한 리팩터링에서는 아래 1줄을 함께 제시:

- `Refactor Check:` 인자 증가 근거 / 구경로 상태(제거·비활성화·마이그레이션 계획)
