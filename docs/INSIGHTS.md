# Structure First - Pre-Skill Insights

이 문서는 `structure-first`를 스킬로 만들기 전에 정리해두었던 코딩 습관/구조 감각을 기록한 메모입니다.
향후 `structure-first-docs` 같은 파생 스킬을 설계할 때 기준점으로 사용합니다.

[한국어](INSIGHTS.md) | [English](INSIGHTS.en.md)

## 1) 읽는 순서와 흐름

- 가장 중요한 것(외부 노출/핵심 성공 경로)부터 위에 둔다.
- 위에서 아래로 읽으면 `Primary Flow -> 세부`로 자연스럽게 내려가야 한다.
- 분기/예외는 메인 흐름을 끊지 않도록 early return 또는 하단 처리로 밀어낸다.

## 2) 분해 감각 (함수/모듈 나누기)

- 분해 자체가 목적이 아니다. 읽힘이 좋아질 때만 분해한다.
- 과도한 함수/파일 분리로 유틸이 난립하지 않게 한다.
- 책임/경계가 실제로 바뀌는 지점에서만 나눈다.

## 3) 조합 가능성 (재사용 방식)

- 범용 유틸 남발보다 조합 가능한 원자(primitive) 세트를 만든다.
- 원자는 I/O가 명확하고 가능하면 순수 함수로 둔다.
- 키 생성/정책 같은 규칙은 바깥으로 빼서 조합성과 유연성을 높인다.

## 4) 상태/컨텍스트에 대한 태도

- 클래스/`this` 의존을 줄이고 순수 함수 기반을 선호한다.
- 부작용은 경계(오케스트레이터/I/O)에 모으고, 내부는 계산/판단 중심으로 둔다.
- 캐시/전역 상태는 정책과 경계를 분리해 다룬다.

## 5) 네이밍 감각

- 약어를 남발하지 말고 문맥을 활용해 짧고 고정된 의미를 유지한다.
- 동사 기반 이름으로 "무슨 일을 하는지"가 바로 드러나야 한다.
- 이름이 설명문처럼 길어지면 구조가 어긋났다는 신호로 본다.

## 6) 코드 밀도/포매팅 (가독성 기준)

- 불필요한 줄바꿈은 줄이고, 한 줄로 충분하면 한 줄로 유지한다.
- `const` 묶음 줄바꿈은 여러 변수를 함께 다룰 때만 허용한다.
- 삼항 연산자도 읽힘이 유지되면 사용한다.
- 리팩터링 시 가능하면 diff를 최소화하고 전체 재작성은 지양한다.

## 7) 테스트 철학 (원자 단위 + 테스트 코드 가독성)

- 원자 단위 분해를 했다면 가능한 한 유닛 테스트를 충분히 붙인다.
- 구현 추종보다 계약(`입력 -> 출력`, 불변조건, 경계값) 검증에 집중한다.
- 테스트 코드도 읽히게 유지한다.
  - `each`/테이블 케이스로 중복을 줄인다.
  - 작은 헬퍼/팩토리는 허용하되 난립은 피한다.
  - 한 테스트에서 한 가지 핵심 주장을 분명히 드러낸다.
  - 케이스 이름을 활용해 데이터 맥락을 보존한다.

## 중간 정리: 통찰을 스킬로 압축한 작업 메모

- 목적을 먼저 고정한다: 변경의 목적을 한 문장으로 쓴다.
- 읽히는 성공 흐름을 먼저 만든다: top-down으로 한 번에 읽히게 만든다.
- 경계는 최소한으로 정의한다: I/O · 판단 · 변환을 구분하되 과증식시키지 않는다.
- 분해는 수단이지 목표가 아니다: 읽힘이 좋아질 때만 분해한다.
- 원자는 역할이 고정된 단위다: I/O 명확성 + 가능하면 순수 함수.
- 조합 책임은 한 곳에 둔다: orchestrator 중심으로 조합하고 원자 간 직접 의존은 줄인다.
- 부작용은 경계로 모은다: 내부 로직은 계산/판단에 집중한다.
- 재사용은 조합으로 얻는다: 미래 가정 추상화는 피한다.
- 테스트는 원자 단위로 충분히 작성한다: 구현이 아니라 계약을 검증한다.
- 테스트 코드도 읽혀야 한다: table case + 절제된 헬퍼로 간결하게 유지한다.

## 문서 스킬 확장 힌트 (`structure-first-docs`)

- 컨벤션 백과사전보다, 짧은 규칙 + 실행 순서 + 체크리스트 형태를 우선한다.
- 문서도 `Primary Reader Flow`를 따르되, 고정 템플릿을 강제하지 않는다.
- `Intent -> Decision -> Steps -> Risks -> Open Questions`는 자주 유용한 기본 흐름 예시로 본다.
- 리뷰 응답은 항상 `이슈/리스크 먼저`를 기본 계약으로 둔다.
